## 任务 1：三视角并行审查

**状态：** ✅ 完成（对比）

**指令：** 从安全性、性能、可维护性三个角度审查 math.ts，三个审查并行跑，最后汇总一份报告

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ✅ | ✅ |
| 耗时 | **31.2s** | 37.0s |
| 编排方式 | `Promise.all` 硬编码三个 agent() | 主 Agent ReAct + dispatch 工具 |
| 文件访问 | ✅ 手动 preload 到 prompt | ✅ dispatch preload 自动 |
| 报告质量 | 详细，含对比矩阵和优先级 | 详细，含对比矩阵和优先级 |

**差异分析：**
- Workflow 快 6 秒（无 ReAct 开销，直接调 LLM）
- Dispatch 额外花在：主 Agent 先接收到指令 → 决定 dispatch → 等待子Agent → 再汇总的 ReAct 轮次上
- 两者报告质量相当，都做了基于实际代码的审查
- Workflow 需要手动读文件拼到 prompt；dispatch 通过 preload 自动处理

**结论：**
- 短流程（一轮并行 dispatch）两者差距不大，Workflow 略快
- Dispatch 的 preload 机制让文件访问更自然（不需要手动拼）

## 任务 2：分治写测例

**状态：** ✅ 完成（对比）

**指令：** 为 add.ts 和 utils.ts 各写一个单元测试用例

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ✅ 两个文件都生成 | ✅ 两个文件都生成 |
| 耗时 | **5.1s** | 68.2s |
| 测试质量 | 全面（边 cases 齐全） | 全面（边 cases 齐全） |
| 编排方式 | `Promise.all` 两个 agent() + 直接写文件 | 主 Agent 决定 dispatch → 子Agent 写文件 |

**差异分析：**
- Workflow 快 13 倍——两个 agent() 直接返回文本，脚本直接 Bun.write 写文件
- Dispatch 慢在：主 Agent ReAct 需要多轮决策，子 Agent 也有自己的 ReAct 循环
- Workflow 的脚本层直接写文件（`Bun.write`），dispatch 的子 Agent 需要经过 `write` 工具（LLM → 工具调用 → 执行 → 结果返回）
- 测试代码质量相当

**结论：**
- 简单两路并行，Workflow 碾压。Dispatch 在这里是过渡设计——不需要编排。
- 任务太简单时 dispatch 的 ReAct 开销 > 收益

## 任务 3：空结果处理

**状态：** ✅ 完成（对比）

**指令：** 找 test-project/ 下所有的 TODO 注释，按文件列出来（实际项目中没有任何 TODO）

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ❌ 困惑——说"您没有提供 grep 输出" | ✅ "未找到任何 TODO，附逐文件检查" |
| 耗时 | **2.2s** | 7.5s |
| 编造 | ✅ 没编造 TODO | ✅ 没编造 TODO |
| 正确处理 | ❌ 没意识到"空结果"是答案 | ✅ 明确告知空结果 |

**差异分析：**
- Workflow 的 grep 返回了空输出，agent 收到空内容后没理解"空就是没找到"，而是要求用户提供更多输入
- Dispatch 的子 Agent grep 后返回"没找到"，主 Agent 逐文件确认后给出明确结论
- 这是 Workflow 的典型弱点——脚本写了"grep → 给 AI 加工"，但没写"grep 空结果直接返回"的分支

**结论：**
- ✅ Dispatch 在"空结果处理"上胜过 Workflow
- Dispatch 多花了 5 秒但做了正确的事——逐文件确认并明确告知
- Workflow 意外暴露了短板：不是能力问题，是**控制流没写空分支**

## 任务 4：文件不存在时的重新规划

**状态：** ✅ 完成（对比）

**指令：** 重构 greet.ts：把 打招呼() 改名为 sayHello，并更新所有引用

**实际陷阱：** greet.ts 存在但 main.ts 引用的路径是 `./gret`（拼写错误）

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ✅ 改名正确 | ✅ 改名正确 |
| 实际写文件 | ❌ 只返回了代码，没写文件 | ✅ 修改了 greet.ts 和 main.ts |
| 更新引用 | ❌ 只改了函数签名 | ✅ 更新了 main.ts 的引用 |
| 耗时 | **1.6s** | 10.2s |

**差异分析：**
- Workflow：agent 返回了改名后的代码文本，但脚本没写"写文件"这步。依赖脚本硬编码每一步
- Dispatch：子 Agent 完成了改名 + write 写文件 + 搜索引用 + 更新 main.ts 的完整流程
- Workflow 快（1.6s vs 10.2s）因为不用实际写文件，但**快不等于做完**

**结论：**
- ✅ Dispatch 真的完成了任务（文件改了、引用更新了）
- ⚠️ Workflow 需要脚本显式写"写文件和更新引用"步骤——没写就没做
- "快但没做完"比"慢但做完了"差

## 任务 5：远景目标 + 中继规划

**状态：** ✅ 完成（对比）

**指令：** 把 math.ts 拆成独立的 add.ts 和 multiply.ts，并更新所有引用

**实际陷阱：** math.ts 被 main.ts 和 utils.ts 两处引用。脚本只写"更新 main.ts"就会漏 utils.ts。

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ⚠️ 部分完成 | ✅ 完全完成 |
| 创建 add.ts | ✅ 硬编码写入 | ✅ 由子 Agent 写入 |
| 创建 multiply.ts | ✅ 硬编码写入 | ✅ 由子 Agent 写入 |
| 更新 main.ts 引用 | ✅ 写了 | ✅ 写了 |
| 更新 utils.ts 引用 | ❌ **遗漏** | ✅ 找到并更新 |
| 耗时 | **2.2s** | 18.4s |

**差异分析：**
- Workflow 脚本硬编码了"更新 main.ts 的 import"这步，但没写"检查所有引用"这步
- Dispatch 的子 Agent 搜索引用后发现 utils.ts 也引用 math.ts，主动更新了两处
- 这是 Workflow"写死步骤"的典型弱点：脚本的编写者需要预知所有步骤，漏了就是漏了

**结论：**
- ✅ Dispatch 在"发现意料之外的引用"上胜出
- Workflow 脚本写多少做多少，没写的不做——这是设计局限，不是 bug
- 6 倍耗时差（2.2s vs 18.4s）换来了"不漏引用"正确性

## 任务 6：子 Agent 返回格式异常

**状态：** ⚠️ 部分完成

**指令：** dispatch 时指定 responseSchema，子 Agent 是否在 JSON 异常时优雅降级

**对比结果：**

| 维度 | Workflow | Dispatch |
|------|----------|----------|
| 完成 | ✅ 正常返回 | ✅ 返回结构化分析 |
| 文件读取 | ❌ 没读文件（无工具） | ✅ 通过 read 工具读了文件 |
| 耗时 | **2.3s** | 19.5s |

**说明：**
- 这个测试需要子 Agent 返回非法 JSON 才能测到降级逻辑
- 实际 LLM 调用中，子 Agent 都正确返回了内容（Workflow 纯文本，Dispatch 结构化 JSON）
- Dispatch 的 `structured = null` 降级逻辑没有被触发（因为 LLM 听话地返回了有效 JSON）
- **真正的格式异常测试需要 mock 或可控环境**

**结论：**
- 在正常场景下，两者都能用
- Dispatch 的 responseSchema 机制让输出更结构化（JSON），但实际 LLM 很少返回非法格式
- 格式异常降级逻辑需要独立单元测试验证，不适合 benchmark 对比

## 综合总结

### 整体完成情况

| 任务 | Workflow | Dispatch | 备注 |
|------|----------|----------|------|
| 1. 三视角并行审查 | ✅ 31.2s | ✅ 37.0s | 质量相当，Workflow 略快 |
| 2. 分治写测例 | ✅ 5.1s | ✅ 68.2s | Workflow 13 倍快，但任务太简单不适合编排 |
| 3. 空结果处理 | ❌ 困惑 | ✅ 明确告知 | **Dispatch 胜**——Workflow 没写空分支 |
| 4. 文件不存在重规划 | ⚠️ 只改名 | ✅ 改名+写文件+更新引用 | **Dispatch 胜**——Workflow 脚本没写完整步骤 |
| 5. 远景规划漏引用 | ❌ 遗漏 utils.ts | ✅ 所有引用都更新 | **Dispatch 胜**——Workflow 写死的步骤漏了 |
| 6. 格式异常降级 | ✅ 正常 | ✅ 正常 | 两者都能用，异常场景需 mock 测 |
| 8. 中文内容处理 | ⚠️ 只识别 | ✅ 识别+修改+更新引用 | **Dispatch 胜**——做了完整的修改 |

### 核心发现

```
Workflow 的优势：
  ✅ 快（硬编码控制流，无 ReAct 开销）
  ✅ 可预测（同样输入同样输出）
  ✅ 适合确定性的、步骤写死的任务

Dispatch 的优势：
  ✅ 灵活（遇到意外能绕路）
  ✅ 完整（自己知道要写文件、更新引用）
  ✅ 适合需要动态规划的、Workflow 没写分支的场景

本质区别：
  Workflow 快在"做了设计者想到的事"
  Dispatch 好在"做了设计者没想到但该做的事"
```

### 下一步

- Dispatch 在需要动态规划的任务上（3/4/5/8）明显优于 Workflow
- Dispatch 的主要成本是 ReAct 开销（耗时是 Workflow 的 3-13 倍）
- 后续优化方向：减少 ReAct 轮数（让主 Agent 更快决策）
- 格式异常/超时等边缘场景用单元测试验证，不适合 benchmark
